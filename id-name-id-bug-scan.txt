[HIGH] BUG-A01: deleteService removes references across all projects by name/id
File: lib/actions.ts:1624-1652
Evidence:
- deleteService(id) resolves serviceName, then runs:
  ProjectModel.updateMany({ agencyId, services: { $in: [id, serviceName] } }, $pull...)
- It is not scoped to serviceToDelete.projectId.
Impact:
- Deleting one service can remove similarly named service references from unrelated projects.
- serviceConfigs are also pulled globally by name/id, causing finance config loss in other projects.

[HIGH] BUG-A02: deleteService safety check blocks by category name globally
File: lib/actions.ts:1632-1639
Evidence:
- TaskModel.countDocuments({ agencyId, category: serviceName }) has no projectId scope.
Impact:
- If same service name exists in multiple projects, deleting service from one project can be blocked by tasks from other projects.

[HIGH] BUG-A03: updateService rename propagates by oldName across all projects
File: lib/actions.ts:1677-1711
Evidence:
- TaskModel.updateMany({ agencyId, category: oldName }, ...)
- ProjectModel.updateMany({ agencyId, services: oldName }, ...)
- ProjectModel.updateMany({ agencyId, 'serviceConfigs.name': oldName }, ...)
- No project-level scope during rename propagation.
Impact:
- Renaming a service in one project can rename categories/service configs in unrelated projects sharing same oldName.
