AI BLOGGER UI + WORKFLOW AUDIT
Date: 2026-03-26

PURPOSE
Check whether the current AI Blogger UI matches the intended product features and workflow.

SUPER-ADMIN CLARIFICATION
The "super-admin" referenced in this audit is the platform/system superadmin area, not a normal agency workspace admin.

Current role split:
- agency admin uses /dashboard/ai-blogger
- platform superadmin uses /super-admin/ai-blogger

Evidence:
- lib/ai-blogger-superadmin.ts requires role "superadmin"
- lib/ai-blogger-dashboard.ts redirects superadmins away from the normal agency AI Blogger dashboard

So:
- this is not an "agency superadmin" workflow
- it is the DC platform-level superadmin UI that manages AI Blogger across agencies

FINAL VERDICT
The AI Blogger UI is strong and mostly aligned on the main agency-side workflow, but it is not a perfect match yet.
The biggest problems are:
- some super-admin AI Blogger routes are dead redirects
- some implemented backend capabilities are not surfaced in UI
- one configured feature area is effectively not wired into the actual workflow
- the visual style is more glass/gradient-heavy than the original "practical DC dashboard" guideline

1A. PLATFORM SUPER-ADMIN UI + FUNCTION CHECK
======================================================================

This section is specifically for the DC platform superadmin area.

WHAT THE PLATFORM SUPER-ADMIN UI CURRENTLY DOES WELL
- lets superadmin open the dedicated AI Blogger admin from the main superadmin sidebar
- shows a cross-agency list of agencies
- shows plan/status/user counts on the agency list
- now shows AI Blogger published count, refresh-candidate count, and top refresh signal summary on the agency list
- lets superadmin open a single-agency AI Blogger admin panel
- lets superadmin review and save agency-specific AI Blogger configuration
- lets superadmin remove dedicated AI Blogger config so the agency falls back to general AI config
- lets superadmin inspect readiness of trends, crawl, SERP, grounded research, Search Console, page performance, image generation, and publish rules
- lets superadmin review an agency refresh queue summary from stored Search Console snapshots

PLATFORM SUPER-ADMIN FUNCTIONS ACTUALLY IMPLEMENTED
These are the meaningful AI Blogger superadmin functions found in code:
- getAllAgenciesWithStats
- getAgencyDetails
- getAgencyAIConfigSuperAdmin
- getAgencyAIBloggerConfigSuperAdmin
- getAgencyAIBloggerOverviewSuperAdmin
- updateAgencyAIBloggerConfigSuperAdmin
- removeAgencyAIBloggerConfigSuperAdmin

Relevant files:
- lib/actions/super-admin.ts

IMPORTANT LIMITATION
The current platform superadmin area is primarily:
- configuration admin
- agency selection
- readiness review
- refresh queue visibility

It is NOT currently a full content-operations interface.

PLATFORM SUPER-ADMIN FUNCTIONS NOT FOUND
I did not find superadmin AI Blogger functions for:
- generating blog drafts as superadmin
- opening a true cross-agency post queue
- editing post content as superadmin
- changing post workflow status as superadmin
- publishing posts as superadmin
- refreshing posts from performance as superadmin
- creating schedules as superadmin
- manually running schedules as superadmin from the UI
- manually running Search Console sync from the UI

Impact:
- the platform superadmin can govern configuration, but cannot fully operate the editorial workflow from the superadmin area
- the actual content workflow still belongs mainly to the agency admin side

2. MISMATCHES / REQUIRED UI WORK
======================================================================

1. UI STYLE DOES NOT FULLY MATCH THE ORIGINAL DC DASHBOARD GUIDELINE
Plan guideline in AI_BLOGGER_PLAN.md:
- "Match existing DC dashboard UI, not the old glassmorphism app."
- "Keep layouts practical and editorial, not flashy."

Current reality:
- the module uses custom glass cards, gradients, glow states, and stronger visual treatment than the normal DC dashboard
- this is not broken, but it does not fully match the original UI direction

Relevant file:
- components/ai-blogger/AIBloggerPrimitives.tsx

Impact:
- stylistic mismatch, not a workflow blocker

2. SUPER-ADMIN ROUTE TREE DOES NOT MATCH WHAT THE FOLDER STRUCTURE SUGGESTS
These routes exist as files but are not real screens:
- app/super-admin/ai-blogger/generate/page.tsx
- app/super-admin/ai-blogger/posts/page.tsx
- app/super-admin/ai-blogger/posts/[slug]/page.tsx
- app/super-admin/ai-blogger/settings/page.tsx

Current behavior:
- each route just redirects back to /super-admin/ai-blogger

Impact:
- dead-end route set
- route tree suggests deeper super-admin workflow pages, but the UI is actually config-first only
- this is the clearest dead-route mismatch in the current AI Blogger UI

3. SEARCH CONSOLE PERFORMANCE SYNC EXISTS, BUT THERE IS NO VISIBLE UI CONTROL FOR IT
Implemented backend:
- secured sync endpoint exists at app/api/ai-blogger/sync-performance/route.ts
- performance data powers the post detail performance loop and refresh queue

Missing in UI:
- no "sync now" button
- no last global sync status
- no failure/success status surface for editors or super-admins

Impact:
- the performance workflow is real, but operational control is hidden
- refresh queue quality depends on background syncs the UI does not expose

4. SCHEDULE AUTOMATION UI IS ONLY PARTIALLY COMPLETE
Current settings UI supports:
- create schedule
- activate/pause schedule
- next run / last run display

Missing from UI:
- edit existing schedule
- delete existing schedule
- manual run now
- detailed run history / per-schedule execution history
- surfaced schedule failures/retries

Relevant files:
- components/ai-blogger/AIBloggerSettingsWorkspace.tsx
- app/api/ai-blogger/run-schedules/route.ts

Impact:
- workflow is usable, but schedule operations are still partial from a UI/ops perspective

5. PAGE PERFORMANCE / PAGESPEED CONFIG IS EFFECTIVELY A DEAD FEATURE IN THE CURRENT UI FLOW
Current UI:
- super-admin config exposes page performance provider, API key, threshold, strategy

Current implementation check:
- in lib/actions/ai-blogger.ts the pagePerformance key is decrypted
- no meaningful generator/editor/dashboard usage was found for this feature area

Relevant files:
- components/super-admin/AIBloggerAgencyConfigClient.tsx
- lib/actions/ai-blogger.ts

Impact:
- config exists
- UI makes it look active
- workflow usage is not actually visible
- this is a real mismatch between surfaced admin controls and delivered product behavior

6. SUPER-ADMIN CAN SEE SUMMARY OPS, BUT NOT FULL CONTENT OPS
Current super-admin state:

Missing:
- no real super-admin posts queue screen

5. BOTTOM LINE
======================================================================

The current AI Blogger UI is not broken and the core agency workflow is genuinely good.
For the main agency user journey, it is mostly aligned with the product features.

But it is not a perfect match yet because:
- some super-admin routes are fake/dead
- the platform superadmin area is config-strong but workflow-light
- some backend automation is hidden from the UI
- one admin feature area (page performance) looks more complete in config than it is in real usage
- some operational workflow surfaces are still partial
