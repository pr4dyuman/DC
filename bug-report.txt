PHASE 1: AUTHENTICATION & AUTHORIZATION
================================================================================

[CRITICAL] BUG-001: Marketing Admin Auth Uses Hardcoded Static Token ✅ FIXED
  File: lib/authMiddleware.js (line 10-11)
  File: app/api/auth-dc/login/route.js (line 42-45)
  Description: The marketing admin authentication uses a HARDCODED static
    cookie value 'logged_in_secret_value' as the auth token. Anyone who knows
    this value can forge the cookie and gain admin access to the marketing
    panel (blog, categories, testimonials).
  Impact: Full unauthorized access to CMS/marketing admin panel.
  Fix: Replace with JWT-based auth like the main platform uses.

[CRITICAL] BUG-002: Marketing Admin Login Stores Passwords in Plain Text ✅ FIXED
  File: app/api/auth-dc/login/route.js (line 30-33)
  Description: The marketing admin login compares passwords with plain text
    equality (`admin.password !== password`). On first login, it even
    creates the admin account with the raw password stored in MongoDB.
    No bcrypt hashing is used.
  Impact: Passwords exposed if database is compromised.
  Fix: Use bcrypt hashing (like the main auth system in lib/auth.ts).

[CRITICAL] BUG-003: Marketing Admin Auto-Creates Account on First Login ✅ FIXED
  File: app/api/auth-dc/login/route.js (line 25-35)
  Description: If no admin exists with the hardcoded email, any user can
    create an admin account by providing any password. This is essentially
    an open registration for admin accounts.
  Impact: Unauthorized admin account creation.
  Fix: Remove auto-creation; require proper seeding/setup.

[HIGH] BUG-004: Legacy Cookie Fallback Still Active in agency-context.ts ✅ FIXED
  File: lib/agency-context.ts (line 37-39)
  Description: getCurrentAgency() falls back to reading raw 'userId' from
    cookies if JWT session is not found. This means if someone manually set
    a 'userId' cookie to another user's ID, they could impersonate that user
    for agency resolution (though not for auth checks in actions.ts which
    use JWT-only via getSessionUser).
  Impact: Potential agency context confusion for unauthenticated requests.
  Fix: Remove the legacy cookie fallback entirely.

[HIGH] BUG-005: Legacy Cookie Fallback in getCurrentUser()
  File: lib/actions.ts (~line 910-935)
  Description: getCurrentUser() has a legacy fallback after the JWT check
    that reads from raw cookies (`getSessionId()`). This second path could
    allow session confusion since it checks all 3 collections
    (SuperAdmin, User, Client) without JWT verification.
  Impact: Potential privilege confusion.
  Fix: Remove the legacy fallback block after the JWT session check.

[MEDIUM] BUG-006: Rate Limiting is In-Memory, Per-Process Only
  File: lib/auth.ts (line 10-13), app/api/signup/route.ts, app/api/signup/send-otp/route.ts
  Description: Rate limiting uses in-memory Maps. In a multi-server or
    serverless deployment (Vercel), each function invocation gets a fresh
    memory space, making rate limiting completely ineffective.
  Impact: Rate limiting doesn't work in production (serverless/multi-instance).
  Fix: Use Redis, Upstash, or a database-backed rate limiter.

[MEDIUM] BUG-007: OTP Store is In-Memory, Lost Between Requests
  File: app/api/signup/send-otp/route.ts (line 8)
  Description: OTP codes are stored in an in-memory Map. In serverless,
    the OTP may be generated by one instance and verified by another,
    causing OTP verification to always fail.
  Impact: Signup OTP flow broken in serverless deployments.
  Fix: Store OTPs in MongoDB or Redis with TTL.

[MEDIUM] BUG-008: No Next.js Middleware for Route Protection
  Description: There is no middleware.ts file in the project. Route
    protection happens only at the layout/page level. This means:
    - API routes under /api/ that forget auth checks are exposed
    - There's no centralized auth gate; each route must self-protect
    - No CSRF protection at the middleware level
  Impact: Easy to miss auth checks on new routes.
  Fix: Add middleware.ts with route-based auth checks.

[MEDIUM] BUG-009: authenticateUser() Doesn't Validate Email Input
  File: lib/auth.ts (line 108)
  Description: The authenticateUser function calls checkLoginRateLimit(email)
    but never validates/sanitizes the email input before using it in DB queries
    (SuperAdminModel.findOne({ email }), UserModel.findOne({ email }), etc.)
  Impact: Potential NoSQL injection if email contains MongoDB operators.
  Fix: Add validateEmail(email) call before DB queries.

[LOW] BUG-010: Login Rate Limit Memory Leak
  File: lib/auth.ts (line 10)
  Description: The loginAttempts Map is never cleaned up. Old entries
    accumulate indefinitely, causing memory growth over time.
  Fix: Add periodic cleanup (e.g., clear expired entries every 30 minutes).

[LOW] BUG-011: Cookie maxAge Missing on Legacy Cookies
  File: lib/auth.ts (line 72-73)
  Description: Legacy cookies ('userId', 'userRole') don't have maxAge set,
    so they become session cookies (cleared when browser closes), while
    auth_token has a 24h maxAge. This inconsistency means the JWT token
    persists but legacy cookies don't.
  Fix: Add maxAge: 60 * 60 * 24 to legacy cookies for consistency.

PHASE 2: DATABASE & DATA INTEGRITY
================================================================================

[HIGH] BUG-012: db.get() Fetches ALL Data from ALL Collections
  File: lib/db.ts (line 93-110)
  Description: The db.get() cached function fetches EVERY document from ALL
    15 collections in parallel. For agencies with thousands of records, this
    loads everything into memory on every request where db.get() is called.
  Impact: Extreme memory usage and slow responses for large agencies.
  Fix: Remove or deprecate db.get(); use targeted queries only.

[HIGH] BUG-013: db.update() Full-Collection Diff is Extremely Expensive
  File: lib/db.ts (line 155-200)
  Description: db.update() reads ALL data, applies a callback, then diffs
    the entire dataset to find changes. This means every mutation via
    db.update() loads + compares all records across all collections.
  Impact: O(n) cost per update, where n = total records across all collections.
  Fix: Use direct MongoDB operations instead of db.update().

[MEDIUM] BUG-014: Missing agencyId Filter When Agency is Null
  File: lib/actions.ts (multiple locations)
  Description: Many functions use `const agencyFilter = agency ? { agencyId: agency.id } : {};`
    When agency is null (e.g., no agency context), the filter becomes {},
    meaning the query returns ALL data across ALL agencies. This happens when:
    - User's JWT doesn't have agencyId
    - Super admin hasn't selected an agency
    - Agency lookup fails silently
  Impact: Cross-tenant data leakage when agency context is missing.
  Fix: Throw an error or return empty results when agency is null for
    tenant-scoped operations (not super-admin operations).

[MEDIUM] BUG-015: getAllUsers() and getAllClients() Lack Role Check
  File: lib/actions.ts (~line 231-240)
  Description: getAllUsers() and getAllClients() don't call requireRole()
    or requireAuth(). Any authenticated user (including clients) can call
    these server actions and get full user/client lists.
  Impact: Clients can enumerate all employees and other clients.
  Fix: Add requireRole('admin', 'manager') checks.

[MEDIUM] BUG-016: getRecentActivity() No Auth Check
  File: lib/actions.ts (~line 420)
  Description: getRecentActivity() doesn't verify the user is authenticated
    or authorized. Any authenticated user can see all agency activity.
  Impact: Employees/clients see admin-level activity logs.
  Fix: Add role-based filtering (employees see only theirs, clients excluded).

[MEDIUM] BUG-017: Orphaned Data When Archiving Clients
  File: lib/actions.ts (~line 1080)
  Description: deleteClient() soft-deletes (archives) the client but doesn't
    update their projects, tasks, or invoices. Projects still reference the
    archived client's ID, and the client's login credentials remain active.
  Impact: Archived clients can still log in and see their projects.
  Fix: Also disable login for archived clients; add archived check in auth.

[MEDIUM] BUG-018: deleteProject() Doesn't Delete Transactions
  File: lib/actions.ts (~line 2175)
  Description: deleteProject() deletes tasks, assets, invoices, and activities
    but does NOT delete associated transactions. Financial records for deleted
    projects persist and still appear in finance stats.
  Impact: Ghost transactions from deleted projects skew financial reports.
  Fix: Add TransactionModel.deleteMany({ projectId: id }) to the cleanup.

[MEDIUM] BUG-019: agencyId Can Be 'default-agency' String
  File: lib/db.ts (line 85), multiple backfillAgencyId calls
  Description: Legacy data without agencyId gets 'default-agency' as a
    fallback. But there's no actual agency with id='default-agency' in the
    database. Queries filtering by this ID return no agency, causing
    permission checks to behave unexpectedly.
  Impact: Legacy data may not be properly scoped or accessible.
  Fix: Create a migration to assign proper agencyIds to all legacy records.

[MEDIUM] BUG-020: No Index on `id` Field (Custom ID, Not MongoDB _id)
  File: lib/mongodb.ts (schemas)
  Description: All queries use `{ id: someValue }` with custom string IDs,
    but Mongoose schemas don't show explicit unique indexes on the `id` field.
    Without indexes, every findOne({ id }) does a full collection scan.
  Impact: Degrading performance as collections grow.
  Fix: Add { id: { type: String, unique: true, index: true } } to all schemas.

[LOW] BUG-021: Activity Log Searches by User Name, Not ID
  File: lib/actions.ts (~line 760, 770)
  Description: getUserActivity() and getEmployeeDashboardData() query
    activities by `{ user: user.name }`. If a user changes their name,
    their old activities become orphaned/unfindable.
  Impact: Name changes break activity history.
  Fix: Store userId in activities and query by ID instead of name.

[LOW] BUG-022: No Cascade Delete for Notifications
  Description: When users/clients are deleted, their notifications remain
    in the database. There's no cleanup of notifications for deleted users.
  Fix: Add NotificationModel.deleteMany({ userId: id }) on user deletion.

[CRITICAL] BUG-023: Upload Route Has No Authentication ✅ FIXED
  File: app/api/upload-dc/route.js (line 1-30)
  Description: The file upload endpoint has NO authentication check.
    Anyone can POST a file and it gets saved to public/uploads/. There's also:
    - No file type validation (can upload .exe, .php, etc.)
    - No file size limit
    - No virus scanning
    - Files are served statically from /public/uploads/
  Impact: Unauthenticated file upload, potential server-side code execution.
  Fix: Add auth check, file type whitelist, size limit, and store outside /public.

[CRITICAL] BUG-024: Upload Path Traversal Vulnerability ✅ FIXED
  File: app/api/upload-dc/route.js (line 16)
  Description: The filename is derived from `file.name.replaceAll(' ', '_')`
    but doesn't sanitize path separators. A filename like "../../../etc/passwd"
    could potentially write files outside the uploads directory.
  Impact: Arbitrary file write on the server.
  Fix: Use path.basename() and strip all path separators from filename.

[HIGH] BUG-025: Contact Form Has No Rate Limiting
  File: app/api/contact-dc/route.js
  Description: The public contact form endpoint has no rate limiting.
    An attacker could spam it to fill the database and trigger excessive
    email sends via Brevo (potentially consuming email quota).
  Impact: Email quota exhaustion, database pollution.
  Fix: Add IP-based rate limiting.

[HIGH] BUG-026: Contact Form Has No Input Sanitization
  File: app/api/contact-dc/route.js (line 8-11)
  Description: Fields (fullName, email, phone, companyName, message) from
    the request body are saved directly to MongoDB without sanitization.
    Only basic presence and email format checks exist. No XSS prevention,
    no length limits, no NoSQL injection protection.
  Impact: XSS via stored user input, NoSQL injection, oversized payloads.
  Fix: Apply sanitizeName(), sanitizeString(), sanitizePhone(), validateEmail().

[HIGH] BUG-027: Blog POST Route Has No Input Sanitization
  File: app/api/blog-dc/route.js (line 35-36)
  Description: Blog creation accepts `await req.json()` and passes the
    entire body directly to `Blog.create(body)` without any sanitization.
    The auth check uses the hardcoded token (BUG-001).
  Impact: XSS via blog content, MongoDB field injection.
  Fix: Validate and sanitize all fields before creating.

[HIGH] BUG-028: Testimonial Route Has No Input Sanitization on Update
  File: app/api/testimonial/route.js (PUT handler)
  Description: Testimonial updates pass user input directly to
    findByIdAndUpdate without sanitization. XSS via testimonial text
    that gets rendered on the public marketing page.
  Impact: Stored XSS on public-facing pages.
  Fix: Sanitize text, name, company fields.

[MEDIUM] BUG-029: Category Route Uses MongoDB _id Directly
  File: app/api/category/route.js (DELETE handler, line ~52)
  Description: The DELETE handler uses `Category.findByIdAndDelete(id)` where
    `id` comes directly from query params without validation. This uses
    MongoDB's _id field, which is inconsistent with the rest of the app
    that uses custom `id` fields.
  Impact: Inconsistent API behavior; potential ObjectId injection.
  Fix: Validate the ID format before using it.

[MEDIUM] BUG-030: Singularity Route Uses User-Supplied userId in Log
  File: app/api/singularity/route.ts (~line 95)
  Description: The route destructures `userId` from the request body and
    uses it in a console.log (line ~95), even though it correctly uses
    `session.userId` for actual operations. The user-supplied userId is
    potentially logged, which could pollute logs.
  Impact: Log injection, confusion in debugging.
  Fix: Remove `userId` from the destructured body entirely.

PHASE 4: AI/SINGULARITY TOOL SAFETY
================================================================================

[HIGH] BUG-031: Singularity Agent Mode Non-Live Doesn't Use Tool Calling
  File: app/api/singularity/route.ts (~line 137-160)
  Description: When using a non-live model in agent mode, the code calls
    `generateContent(aiConfig, fullPrompt, systemInstruction)` which is a
    simple text generation — NOT tool calling. The filtered tools are
    computed but never passed to the AI. Only the live API path uses
    actual tool calling.
  Impact: Non-live models in agent mode can't execute tools (create tasks,
    update projects, etc.). They only generate text responses.
  Fix: Use the generateContentWithChat or tool-calling API for non-live models.

[HIGH] BUG-032: AI Tool Permission Check Doesn't Verify agencyId
  File: lib/singularity-tools.ts (~line 145-160)
  Description: executeTool() checks user role and AI permissions but doesn't
    verify that the userId making the request belongs to the current agency.
    It looks up the user role globally, not scoped to a specific agency.
  Impact: If a user from Agency A somehow calls with a userId from
    Agency B, the role check would pass incorrectly.
  Fix: Add agencyId scoping to getUserRole() lookup.

[MEDIUM] BUG-033: Tool Arguments Not Sanitized Before Execution
  File: lib/singularity-tools.ts (entire executeTool function)
  Description: AI-generated tool arguments (args.name, args.title,
    args.description, etc.) are passed directly to server actions without
    additional sanitization. While the server actions themselves have
    sanitization, the AI could generate arguments that bypass certain
    client-side expectations.
  Impact: AI could create records with unexpected content.
  Fix: Add sanitization layer in executeTool before calling actions.

[MEDIUM] BUG-034: delete_task Tool in Singularity Missing Password Check
  File: lib/singularity-tools.ts (delete_task case)
  Description: The delete_task tool calls deleteTask(args.taskId) directly.
    But the deleteTask action in actions.ts requires checking user
    permissions. If the AI calls delete_task, it bypasses the normal
    UI flow where users confirm deletion.
  Impact: AI can delete tasks without user's explicit confirmation.
  Fix: Add a confirmation step or "dry run" flag for destructive AI operations.

[LOW] BUG-035: Singularity Context Builds N+1 Queries
  File: lib/singularity-context.ts (~line 85-120)
  Description: buildSingularityContext() fetches all projects, then for EACH
    project (up to 20), it calls getTasks(projectId) individually. Similarly,
    for each team member, it calls getUserTasks(userId). This creates an
    N+1 query pattern.
  Impact: Slow context building with many projects/users (20 projects +
    20 users = 40+ additional DB queries).
  Fix: Batch fetch all tasks once, then filter by projectId/userId in memory.

  File: app/dashboard/messages/MessagesPageClient.tsx (~line 68-77)

[MEDIUM] BUG-040: typeof window Check for Client-Side Code

  Impact: Potential silent failures; redundant code.
  Fix: Remove unnecessary typeof window checks in "use client" components.

[LOW] BUG-041: No Empty State Components for Dashboard Sections
  Description: When a new agency has 0 projects, 0 clients, 0 tasks, etc.,

PHASE 6: FINANCE & BUSINESS LOGIC
================================================================================

[HIGH] BUG-042: Revalidation Path Has Extra Spaces (Broken Caching)
  File: lib/actions.ts (~line 2330)
  Description: Two revalidatePath() calls have spaces inside template literals:
    `revalidatePath('/ dashboard / projects / ${transaction.projectId} ');`
    `financeLink: '...http://localhost:3000'} /dashboard/finance'` (line ~2410)
    These paths will never match any actual route, so cache revalidation
    is silently broken after creating transactions and sending emails.
  Impact: Stale data shown after financial operations.
  Fix: Remove extra spaces from the paths.

[HIGH] BUG-043: No Double-Payment Prevention for Salary
  File: lib/actions.ts (payEmployee, ~line 2738)
  Description: payEmployee() creates a salary transaction without checking
    if the employee has already been paid for that month. The check exists
    in getPayrollStatus() for display purposes, but the payEmployee()
    action itself doesn't prevent double-payment.
  Impact: An admin (or AI) could accidentally pay an employee twice.
  Fix: Check for existing salary transaction for the same user+month before
    creating a new one.

[HIGH] BUG-044: Invoice Can Be Marked Paid Multiple Times via updateInvoiceStatus
  File: lib/actions.ts (~line 2540)
  Description: The legacy updateInvoiceStatus() function directly sets
    invoice status without checking current status. An invoice already
    marked as "Paid" can be re-marked as "Paid" or moved to "Pending"
    without any state machine validation.
  Impact: Invoice status can be set to any state from any state.
  Fix: Add state transition validation (e.g., Paid -> Pending not allowed).

[MEDIUM] BUG-045: Refund Amount Validation Has Race Condition
  File: lib/actions.ts (createRefund, ~line 3867)
  Description: createRefund() checks that (existingRefunds + newRefund) <=
    projectIncome, but this check is not atomic. Two concurrent refund
    requests could both pass the check before either writes, resulting
    in total refunds exceeding income.
  Impact: Refunds can exceed project income under concurrent requests.
  Fix: Use MongoDB findOneAndUpdate with conditions, or implement
    optimistic locking.

[MEDIUM] BUG-046: No Validation on Transaction Amount Maximum
  File: lib/actions.ts (createTransaction, ~line 2249)
  Description: createTransaction() checks `amount > 0` but doesn't use
    validateAmount() which caps at 999,999,999. Extremely large amounts
    could be entered.
  Impact: Unrealistic financial figures; potential overflow in charts.
  Fix: Use validateAmount() for all financial inputs.

[MEDIUM] BUG-047: Monthly Invoice Generation Can Create 0 Invoices
  File: lib/actions.ts (~line 1320-1340)
  Description: When creating a project with monthly billing, the code
    calculates `monthsDiff` using simple subtraction. If the due date
    is before the billing start date, monthsDiff could be negative or zero,
    and `Math.min(monthsDiff, 12)` could result in 0 invoices — no error.
  Impact: Silent failure to create expected invoices.
  Fix: Validate that billing start date is before project due date.

[LOW] BUG-051: Client Welcome Email Also Sends Plain Text Password
  File: lib/actions.ts (createClient, ~line 1985)

  Impact: Minor notification spam.
  Fix: Deduplicate notification recipients.

[HIGH] BUG-053: Chat Polling Interval Different in Mobile View
  File: app/dashboard/messages/MessagesPageClient.tsx (line 28-29)
  File: components/chat/ChatOverlay.tsx (line 27)
  Description: Both the full messages page and the chat overlay component
    maintain independent state for mobile/desktop views (mobileView state).
    The contacts polling (120s) and messages polling (10s) run identically
    regardless of mobile vs desktop. However, on mobile, switching between
    'sidebar' and 'chat' views causes activeContactId to change frequently,
    which triggers message reloads. On slow mobile connections, this creates
    a cascade of overlapping fetch requests.
  Impact: Excessive network requests on mobile; potential stale data display.
  Fix: Debounce contact selection on mobile; cancel previous fetch on new selection.

[MEDIUM] BUG-054: Mobile Sidebar Completely Hidden via CSS, Not Conditional
  File: components/layout/AuthenticatedLayout.tsx (line 44)
  Description: The sidebar is hidden on mobile with `hidden md:flex`. This
    means on mobile, the sidebar never renders, and the Sidebar component
    (with its data fetching) still executes on the server even though
    it's hidden. More importantly, there's no mobile navigation alternative
    visible — users must reach md breakpoint to see navigation.
  Impact: No visible navigation on small mobile screens (< md breakpoint)
    unless the Topbar provides a hamburger menu.
  Fix: Verify Topbar has mobile menu; if not, add mobile navigation drawer.

[MEDIUM] BUG-055: Chat Overlay Duplicated State with Messages Page
  File: components/chat/ChatOverlay.tsx vs app/dashboard/messages/MessagesPageClient.tsx
  Description: Both components implement IDENTICAL chat logic but
    independently. They maintain separate contact lists, message lists,
    polling intervals, and state. If a user is on the messages page with
    the overlay open, TWO sets of polling happen simultaneously,
    and messages may appear in one but not the other.
  Impact: Double API calls, inconsistent message state.
  Fix: Share a single ChatContext/store for all chat UI surfaces.

[MEDIUM] BUG-056: Heartbeat Not Reset on Mobile App Switch/Background
  File: lib/chat.ts (heartbeat function)
  Description: The heartbeat mechanism updates lastActiveAt for online
    status, but there's no handling for mobile browsers going to background.
    When a user switches apps on mobile, the heartbeat timer keeps running
    (or stops depending on the browser). The 1-minute online threshold
    doesn't account for mobile wake/sleep patterns.
  Impact: Mobile users shown as "online" when they're not; or shown as
    "offline" when they're actively using the app in background.
  Fix: Use visibilitychange event to pause/resume heartbeat.

[LOW] BUG-057: No Touch-Friendly Interactions for Drag-and-Drop
  Description: The project uses @dnd-kit for drag-and-drop (likely for
    task boards). On mobile, drag-and-drop can be tricky with touch events.
    There may be no fallback (like action buttons) for mobile users who
    can't easily drag items.
  Fix: Add mobile-friendly buttons for status changes alongside DnD.

[CRITICAL] BUG-058: Sensitive Environment Variables Exposed in next.config.ts ✅ FIXED
  File: next.config.ts (line 13-18)
  Description: The next.config.ts explicitly adds server-side secrets to the
    `env` block:
      MONGODB_URI: process.env.MONGODB_URI,
      JWT_SECRET: process.env.JWT_SECRET,
      BREVO_API_KEY: process.env.BREVO_API_KEY,
    When placed in `env {}` in next.config, these values get embedded in the
    client-side JavaScript bundle. This exposes your DATABASE CONNECTION STRING,
    JWT SECRET, and EMAIL API KEY to anyone who views your page source.
  Impact: Full database access, JWT token forging, email API abuse.
  Fix: REMOVE all secrets from next.config.ts `env` block. Server-side code
    already has access to process.env. Only NEXT_PUBLIC_* vars should be
    in the env block.

[HIGH] BUG-059: AI_ENCRYPT_KEY Fallback Returns Plaintext
  File: lib/mongodb.ts (encryptApiKey, line 15)
  Description: If AI_ENCRYPT_KEY is not set, encryptApiKey() returns the
    API key as plaintext: `if (!AI_ENCRYPT_KEY) return plaintext;`
    And decryptApiKey() returns the raw value if it doesn't start with 'enc:'.
    This means in development environments, API keys are stored unencrypted.
  Impact: API keys stored in plain text in development DB.
  Fix: Require AI_ENCRYPT_KEY in all environments, or use a warning.

[HIGH] BUG-060: console.log in Production Disabled But console.error Remains ✅ FIXED
  File: next.config.ts (line 9)
  Description: `removeConsole: process.env.NODE_ENV === "production"` removes
    ALL console methods including console.error. This means error logging
    from catch blocks throughout the entire app is silently suppressed in
    production. You'll never see error logs.
  Impact: Zero error visibility in production; debugging impossible.
  Fix: Change to `removeConsole: { exclude: ['error', 'warn'] }`.

[HIGH] BUG-061: getContacts() Fetches ALL Users AND ALL Messages
  File: lib/chat.ts (getContacts, ~line 75-85)
  Description: getContacts() fetches ALL users + ALL clients + ALL messages
    involving the current user. For agencies with many users and message
    history, this query returns unbounded data.
  Impact: Slow chat loading; excessive memory usage.
  Fix: Paginate contacts; only fetch recent messages for last-message preview.

[MEDIUM] BUG-062: MongoDB Connection URL Logged (Even Partially Masked)
  File: lib/mongodb.ts (line 71)
  Description: `console.log('URI:', MONGODB_URI!.replace(...))`
    Even with password masking, the host/port/database name are logged.
    Combined with BUG-060's removeConsole issue, this is either a security
    leak (in dev) or invisible (in prod).
  Fix: Remove the URI log entirely, or only log in development.

[MEDIUM] BUG-063: No Request Timeout on AI API Calls
  File: lib/ai-provider.ts, app/api/singularity/route.ts
  Description: AI provider calls (Gemini, OpenAI, etc.) have no timeout
    configured. If the AI provider is slow or unresponsive, the server
    action hangs indefinitely, consuming server resources.
  Impact: Requests hang; server resource exhaustion.
  Fix: Set timeout on AI API calls (e.g., 30 seconds for text, 60 for tools).

[MEDIUM] BUG-064: Server Actions Body Size 10MB May Be Too Generous
  File: next.config.ts (line 5)
  Description: Server actions body size limit is set to 10MB. This allows
    very large payloads to be sent to any server action, potentially

  Impact: Large payload attacks on server actions.
  Fix: Consider reducing to 2MB and using dedicated upload endpoints for files.

[LOW] BUG-065: getDashboardMetrics() Fetches ALL Transactions
