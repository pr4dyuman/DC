RBAC + PERMISSION MANAGER PLAN

Purpose
This document captures the recommended direction for replacing hard-coded role behavior with a central permission manager, while preserving the existing different dashboard/finance/client experiences in the app.

IMPORTANT CLARIFICATION
This plan does NOT say all users must see the same pages.
In the current project, different users already have different views:
- client has a separate dashboard view
- employee has a personal dashboard view
- admin/manager have management dashboards
- finance also has different logic for client vs admin/manager vs employee

So the correct goal is:
- centralize access control
- keep different UI views where needed
- make module access and permissions configurable
- stop scattering hard-coded role checks across the codebase


1. CURRENT PRODUCT REALITY

Right now the project is an agency operating system, not a fully universal business OS.

It already supports:
- projects
- tasks
- clients
- team
- messages
- finance
- AI
- settings

It already has role-based branching in important places:
- proxy.ts has static route lists by role
- components/dashboard/DashboardContent.tsx renders separate client/admin/employee dashboards
- components/finance/FinanceContent.tsx renders separate finance behavior for client and blocks employee finance access
- lib/actions/access.ts still relies on hard-coded roles
- components/settings/PermissionSettings.tsx only manages a small userPermissions subset

This means the app already uses BOTH:
- access control
- view control

That is normal.


2. FINAL MODEL TO USE

Keep these concepts separate:

A. Platform Identity
- superadmin
- platform-only
- outside all agencies
- manages agencies, trials, AI usage, billing-type platform controls

B. Agency Base Role
- admin
- manager
- employee
- client

These 4 remain in the system, but they should become DEFAULT ACCESS PRESETS, not the only source of truth for security.

C. Job Title
- free text
- editable
- examples: SEO Specialist, HR Lead, Video Editor, Sales Head
- display only
- must never control security

D. User-Level Access Configuration
- enabled modules
- permission values
- optional restrictions
- optional overrides from the base role defaults


3. WHAT YOUR IDEA GETS RIGHT

Your idea is strong because:
- the 4 existing roles give easy starting defaults
- admin can quickly create a user with sensible initial access
- admin can then customize that specific person
- job title stays flexible for any kind of agency team member
- a central permission manager makes changes easier to maintain

This is a good balance between:
- structure
- flexibility
- backward compatibility


4. WHAT MUST BE UNDERSTOOD CLEARLY

Permission manager alone does NOT automatically make the product support every industry.

It helps the project support many kinds of agencies IF those agencies still work roughly like this:
- clients or external stakeholders
- internal team members
- projects or engagements
- tasks/work items
- communication
- files/assets
- billing/finance tracking

Examples it can support well after this work:
- software agency
- marketing agency
- design studio
- video/content agency
- consulting agency
- branding agency
- operations/service agency

Examples that still need deeper product changes beyond permissions:
- hospital workflow
- law case management
- factory/manufacturing
- school/college administration
- warehouse/inventory-heavy business
- construction site operations

Reason:
those need different core entities and workflows, not just different permissions.


5. HOW TO HANDLE "DIFFERENT PAGES FOR DIFFERENT USERS"

This is the most important answer to the concern about dashboard/finance/client differences.

Access control and page rendering should be treated as two different layers:

Layer 1: Access Layer
- decides whether user can open a module/page/action

Layer 2: View Layer
- decides which version of that module/page the user sees

Example:
- two users may both have access to dashboard
- but one sees Admin Dashboard
- another sees Employee Dashboard
- another sees Client Portal Dashboard

So we should not force one single dashboard for everyone.

Instead we should define a VIEW PROFILE.

Recommended initial view profiles:
- admin_view
- manager_view
- employee_view
- client_view

At first these can map directly from the current 4 base roles.

Example:
- baseRole admin -> admin_view
- baseRole manager -> manager_view
- baseRole employee -> employee_view
- baseRole client -> client_view

Later, if needed, view profile can become configurable without breaking permissions.


6. RECOMMENDED ACCESS RESOLUTION FLOW

When the system checks a user, it should resolve access in this order:

1. superadmin check
- if superadmin, use platform auth only
- do not mix superadmin with agency auth

2. base role defaults
- admin default modules/permissions
- manager default modules/permissions
- employee default modules/permissions
- client default modules/permissions

3. module overrides
- turn whole modules on/off for that specific user

4. permission overrides
- fine-grained permissions for that specific user

5. resource scope checks
- own project vs assigned project vs all
- own tasks vs any tasks
- own invoices vs all invoices

6. view profile selection
- choose which UI experience to render

This keeps the system understandable:
- baseRole gives defaults
- permission manager customizes
- viewProfile controls UI variant


7. RECOMMENDED DATA SHAPE

Do not use jobTitle for security.

Recommended user shape:

isSuperadmin: boolean
agencyId: string

baseRole: "admin" | "manager" | "employee" | "client"
jobTitle: string
viewProfile: "admin_view" | "manager_view" | "employee_view" | "client_view"

enabledModules: {
  dashboard: boolean
  projects: boolean
  finance: boolean
  team: boolean
  clients: boolean
  messages: boolean
  settings: boolean
  ai: boolean
}

permissionOverrides: {
  canCreateProject?: boolean
  canManageTasks?: boolean
  canMarkDone?: boolean
  canUseAI?: boolean
  deleteAccess?: "none" | "own" | "any"
}

Later this can grow into:
- custom permission keys
- scoped permissions
- per-agency module catalogs


8. HOW THE 4 DEFAULT ROLES SHOULD WORK

These 4 roles should stay because they are useful presets.

admin
- full agency control
- all modules on
- almost all permissions on
- can manage settings and permissions

manager
- team/process management
- broad operational access
- usually not full agency ownership powers

employee
- task/project work access
- restricted finance/settings/admin controls

client
- client portal only
- limited projects/files/invoices/messages visibility

Important:
these are DEFAULT PRESETS.
Admin can customize per user after assignment.


9. PERMISSION MANAGER DESIGN

Create one central screen in admin settings.

Recommended areas:

A. Module Catalog
- list all modules
- each module contains its permissions

B. Default Role Presets
- admin
- manager
- employee
- client
- allow editing their default module/permission values

C. User Access List
- each row shows person
- base role dropdown
- job title
- module toggles
- permission controls
- reset to base role defaults

D. Optional Custom Presets Later
- duplicate a preset
- save as custom preset
- assign to multiple users

Note:
You can add the + icon beside the role dropdown later for preset creation or preset duplication.
For phase 1, the most important thing is central permission management, not preset UX polish.


10. HOW THIS SUPPORTS MANY TYPES OF AGENCIES

This project can handle many types of agencies because:
- team members can have any job title
- modules can be enabled/disabled per person
- permissions can be customized per person
- base roles provide default starting points
- the same core system can represent different agency structures

Example mappings:

Software agency
- employee -> developer / QA / designer
- manager -> delivery manager / tech lead

Marketing agency
- employee -> SEO executive / content writer / media buyer
- manager -> campaign manager / account manager

Video agency
- employee -> editor / animator / shooter
- manager -> production manager

Consulting agency
- employee -> analyst / associate
- manager -> engagement manager

So the project does not need a separate built-in role for every job type.
That part is handled by:
- base role for defaults
- jobTitle for naming
- permission manager for actual access


11. WHERE CONFLICTS CAN STILL HAPPEN

Your model can still create problems if we do not handle these carefully:

A. Scattered role checks remain in code
- if code still uses role === "admin" everywhere, permission manager becomes fake

B. Middleware and server actions disagree
- hidden buttons are not security
- server-side actions must enforce final permission logic

C. Client data isolation
- client users must only see their own data unless explicitly allowed

D. View profile and permission mismatch
- user may have finance module enabled but still be forced into wrong UI

E. No reset to defaults
- admins will lose track of custom changes


12. SAFE IMPLEMENTATION STRATEGY

Phase 1
- keep current 4 roles
- treat them as default presets
- keep current dashboards/views working
- extend permission manager
- do not break route security yet

Phase 2
- centralize module and permission definitions
- resolve access from preset + overrides
- use same resolver in UI and server actions

Phase 3
- replace static role checks gradually
- dashboard, finance, projects, settings, AI

Phase 4
- replace static route gating in proxy.ts with safer permission-aware checks

Phase 5
- add optional custom presets and improved admin UX


13. BEST PRACTICAL CONCLUSION

Yes, your idea is valid.

Best practical version of it:
- keep superadmin separate
- keep 4 default base roles
- use those roles as default presets
- allow admin to customize modules and permissions per user
- keep jobTitle fully editable
- keep different dashboards/pages for different user experiences
- separate access control from view rendering

This is how the project can support many kinds of agencies without needing a brand new built-in role for every possible team member.

But to be fully honest:
this will make the product much more flexible for many agency types,
not magically universal for every business domain.

That is the correct and realistic target.
