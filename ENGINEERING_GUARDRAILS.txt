AGENCY-OS ENGINEERING GUARDRAILS
Version: 1.0
Scope: Mandatory rules for all future code changes in this repository.

PRE-EDIT GATE (REQUIRED BEFORE ANY CODE CHANGE)
1) Trace authentication and token flow touched by the change.
2) Identify security impact, data impact, UI impact, and workflow impact.
3) Define production-ready approach only; reject temporary workarounds.
4) Confirm compatibility with current architecture, conventions, and runtime.

RULE 1: AUTH AND TOKEN SAFETY
- Preserve existing auth boundaries, session rules, and token lifecycle.
- Never weaken validation, authorization, expiry, rotation, or revocation behavior.
- Avoid exposing tokens, secrets, user-sensitive payloads, or internal errors.
- Any auth-related update must include threat-focused review of affected endpoints and flows.

RULE 2: PRODUCTION-LEVEL QUALITY ONLY
- No temporary patches, bypasses, hidden flags, or fragile shortcuts.
- Implement durable fixes with clear failure handling and stable behavior.
- Keep edge cases and rollback behavior in mind for real traffic conditions.

RULE 3: MONGODB ORGANIZATION AND DATA INTEGRITY
- Keep schema usage consistent, normalized where appropriate, and predictable.
- Preserve indexes, constraints, references, and migration safety.
- Prevent orphaned records, duplicate writes, and inconsistent document structure.
- Database-related changes must not degrade query health or operational maintainability.

RULE 4: CODE ORGANIZATION AND MAINTAINABILITY
- Follow current project patterns for naming, structure, and layering.
- Keep modules cohesive and avoid unnecessary coupling.
- Maintain readability, typed safety where applicable, and clear control flow.

RULE 5: WORKFLOW COMPATIBILITY
- Changes must not break existing product workflows, role-based behavior, or integrations.
- Preserve current API contracts unless an explicit migration path is included.
- Validate backward compatibility for dependent screens, actions, and services.

RULE 6: CONSEQUENCE ANALYSIS BEFORE MERGE
- Check direct and indirect impacts across API routes, UI states, jobs, and data lifecycle.
- Review failure modes, race conditions, and permission boundaries.
- Ensure no existing feature regression is introduced.

RULE 7: THEME CONSISTENCY (BLACK AND WHITE)
- UI updates must work correctly in both dark and light themes.
- Maintain contrast, readability, semantic states, and visual consistency.

RULE 8: RESPONSIVE UI QUALITY
- UI changes must be responsive across mobile, tablet, laptop, and desktop breakpoints.
- Preserve current layout behavior and avoid overflow, clipping, and interaction issues.

RULE 9: BUG REPORTING DUTY
- If any existing bug is discovered during work, report:
  a) where it exists,
  b) expected vs current behavior,
  c) user/business impact,
  d) severity and risk.

RULE 10: RE-VERIFICATION REQUIREMENT
- Re-verify every edit before completion.
- Minimum verification includes:
  1) target behavior check,
  2) regression check for related flows,
  3) lint and typecheck execution where available,
  4) test execution relevant to impacted areas.

MANDATORY EXECUTION ORDER FOR ALL FUTURE CHANGES
Step A: Run Pre-Edit Gate.
Step B: Implement production-grade change.
Step C: Perform consequence analysis.
Step D: Re-verify via lint, typecheck, tests, and impacted flow checks.
Step E: Report discovered bugs and final validation status.
