AI BLOGGER PRODUCTION UI/UX REQUIRED CHANGES
Date: 2026-04-25
Project: DC AI Blogger

VERDICT
The AI Blogger UI is feature-rich, but it is not production-level yet.

Current readiness estimate:
- Agency AI Blogger dashboard: 6/10
- Super-admin AI Blogger area: 4/10
- Public AI Blogger service page: 7/10

The product has many useful workflows, but the UI still feels like an internal build in several places. The biggest issues are unfinished super-admin pages, inconsistent layout density, too many nested rounded cards, placeholder-looking content, cramped mobile states, misleading cluster data, and text that exposes implementation decisions instead of speaking to users.

P0 PRODUCTION BLOCKERS

1. Super-admin AI Blogger pages expose internal scope text
Problem:
- Several super-admin pages say things like "not implemented", "future", "What Is Missing", "Phase Decision", "honest scope", "Queue not implemented", and similar internal planning language.
- This is not acceptable in production. It makes the product look unfinished and exposes internal roadmap decisions.

Affected areas:
- /super-admin/ai-blogger/generate
- /super-admin/ai-blogger/posts
- /super-admin/ai-blogger/settings
- /super-admin/ai-blogger/posts/[slug]

Required changes:
- Remove these nav items until the real workflows exist, or replace them with polished production empty states.
- Do not show internal terms like "scope page", "future implementation", "phase decision", or "not implemented".
- If a feature is not ready, use a clean message like: "This workspace is not available for this role yet."
- Add a clear action: "Open agency workspace", "Select agency", or "View agency configuration".
- Make the super-admin experience feel intentional, not unfinished.


2. Cluster page data is visually and logically misleading
Problem:
- The page shows strong summary numbers such as total clusters, high coverage, or zero orphaned posts.
- The actual cards then show repeated states like "Unnamed Pillar", "Orphaned", and "No cluster assigned".
- This contradiction makes the page feel broken and damages trust.

Affected area:
- /dashboard/ai-blogger/clusters

Required changes:
- Fix the cluster analysis logic so summary metrics match the visible cards.
- Do not display "Unnamed Pillar" as a final user-facing title.
- Group truly unclustered posts into one clear "Unclustered posts" section.
- Use real pillar names, cluster names, or topic group names wherever possible.
- If no cluster exists, show one focused empty state with a CTA to assign or generate clusters.
- Replace repeated orphan cards with a compact table or grouped list.


3. Posts cards are too dense and visually noisy
Problem:
- Cards have too many chips, badges, labels, metadata rows, and decorative thumbnail blocks.
- The placeholder gradient document image feels fake and not production-quality.
- Status badges can clip or truncate, especially on smaller widths.
- The scanning experience is hard because every card competes for attention.

Affected area:
- /dashboard/ai-blogger/posts

Required changes:
- Use a cleaner list or table-first layout for production content management.
- Keep only the top 2 or 3 key signals visible by default: status, keyword, updated date/performance.
- Move secondary metadata into a details panel, hover state, or expanded card.
- Replace fake thumbnails with real featured images. If no image exists, use a small neutral icon state, not a large decorative placeholder.
- Fix badge overflow and truncation.
- Give titles more room and reduce visual competition from chips.
- Make bulk actions clearer and easier to reach.
- On mobile, use compact stacked rows instead of long card blocks.


4. Generate page feels too heavy for first-use
Problem:
- The draft builder exposes too much configuration at once.
- Placeholder text looks like real filled data because examples such as "https://example.com" and recommendation text are visually prominent.
- The pipeline plan uses dense pipe-separated text and feels cramped.
- The main action is far down the page on mobile.

Affected area:
- /dashboard/ai-blogger/generate

Required changes:
- Split the page into clearer steps: Brief, Target, Advanced, Generate.
- Collapse advanced settings by default.
- Lower placeholder contrast so examples do not look entered.
- Replace pipe-separated pipeline text with chips or a vertical stepper.
- Keep the primary generate action visible or sticky on mobile.
- Add inline validation and a clear readiness state.
- Reduce explanatory text inside cards. Use short labels and helpful tooltips.
- Show a preview summary before generation.


5. Overview dashboard is too long and repetitive
Problem:
- The dashboard uses large cards for many metrics and sections.
- Inactive analytics and Search Console areas take too much space even when disabled or empty.
- Recent drafts duplicate the posts workspace and use placeholder thumbnail visuals.
- The page requires too much scrolling before the user understands the next best action.

Affected area:
- /dashboard/ai-blogger

Required changes:
- Convert the top area into a compact operations dashboard.
- Use a small metrics strip instead of oversized cards.
- Show one clear next action: generate draft, review posts, connect data, or refresh content.
- Hide or collapse inactive analytics behind a setup state.
- Replace repeated draft cards with a compact recent activity list.
- Reduce card height and repeated explanatory text.
- Make the dashboard feel like a command center, not a long report page.


6. Post detail page is powerful but overwhelming
Problem:
- The post detail page has too many simultaneous panels, audit blocks, sidebar cards, and repeated checklist items.
- A published or view-only state can still show a "Save Changes" action, which feels incorrect.
- The editor content is visually dense and not comfortable for writing or reading.
- The right sidebar tries to show everything at once.

Affected area:
- /dashboard/ai-blogger/posts/[slug]

Required changes:
- If the post is published or read-only, hide "Save Changes" or replace it with "Create editable version".
- Create a clearer layout hierarchy: editor first, audit second, performance third.
- Collapse long audit checklists by default.
- Use summary cards for SEO/performance with drill-down details.
- Improve editor typography for long content.
- Keep destructive or publishing actions visually separated.
- Make the sidebar sticky only if it remains readable and does not crowd content.
- Add a strong status banner near the title: Draft, Review, Scheduled, Published, Needs refresh.


7. Refresh Queue empty state is not actionable enough
Problem:
- The page can show zero candidates while filters, cards, and metrics remain visible.
- If Search Console is disabled, the user sees empty data but does not get a strong setup path.

Affected area:
- /dashboard/ai-blogger/refresh-queue

Required changes:
- When Search Console is disabled or not configured, show a dedicated setup state first.
- Hide candidate filters until data exists.
- Explain clearly why there are no candidates.
- Add CTA: "Connect Search Console", "Run sync", or "Configure performance source".
- Show skeletons or progress if data is loading.
- Keep the empty state compact and confident.


8. Mobile navigation and mobile cards need major polish
Problem:
- AI Blogger subnav becomes icon-only on mobile.
- Hidden text labels may not be enough for accessibility if links do not have proper aria labels.
- Search, filter, and action controls become cramped.
- Cards become very tall, especially posts and overview sections.

Affected areas:
- All AI Blogger dashboard pages on mobile

Required changes:
- Replace the mobile subnav with a labeled dropdown, bottom sheet, or segmented control.
- Add aria-labels to icon-only links and buttons.
- Make filters collapsible behind a single filter button.
- Convert long cards into compact rows.
- Keep the primary page action sticky or near the top.
- Avoid two-column desktop assumptions leaking into mobile.
- Ensure every button label fits without wrapping awkwardly.


9. Visual system feels too decorative for an operations dashboard
Problem:
- Many cards use large 22px to 28px radii, nested glass effects, shadows, borders, gradients, and all-caps micro labels.
- The UI reads more like a visual experiment than a production content operations tool.
- There are too many card-inside-card structures.

Affected areas:
- Most AI Blogger dashboard components

Required changes:
- Standardize radius to 8px to 12px for operational panels.
- Use larger radii only for modals or high-level marketing surfaces.
- Reduce nested cards.
- Use clean panels, tables, and lists for dense workflows.
- Reduce decorative gradients and glass effects.
- Use color mostly for status, hierarchy, and action.
- Keep all-caps labels only for small metadata, not every section header.
- Make spacing consistent across pages.


10. Public AI Blogger service page still sounds pre-launch
Problem:
- Copy says or implies that the service is future-facing or not fully live.
- The hero relies on a generic illustration instead of showing the actual product or output.
- The page is visually stronger than the dashboard, but it does not yet prove the product experience.

Affected area:
- /services/ai-blogger

Required changes:
- Replace pre-launch language with confident production copy.
- Show real AI Blogger dashboard screenshots, generated article examples, or workflow output.
- Make the offer and outcome obvious in the first viewport.
- Tighten the lower sections and remove any vague future wording.
- Add proof: sample posts, workflow steps, content quality, publishing process, or client-ready result.

TEXT AND COPY CHANGES

Remove or replace these internal phrases:
- "not implemented"
- "future"
- "future scope"
- "scope page"
- "honest scope"
- "phase decision"
- "config-first"
- "old route behavior"
- "redirect loop"
- "Queue not implemented"
- "What Is Missing"

Replace placeholder-sounding labels:
- "Unnamed Pillar" -> "Unclustered posts" or the actual pillar name.
- "No cluster assigned" -> "Not assigned to a cluster yet" with action.
- "No image" -> "Add featured image" or a small neutral image icon.
- "AI inferred" -> "Will be inferred" before generation, or show the actual value after generation.

Make terminology consistent:
- Pick one label for the creation workflow: "Generate Draft" or "Create Draft".
- Pick one label for content list: "Posts", "Content Queue", or "Articles".
- Pick one label for refresh workflow: "Refresh Queue" or "Performance Refresh".
- Use "Search Console" consistently, not mixed variations.
- Use action verbs on buttons: "Generate", "Review", "Publish", "Connect", "Sync", "Refresh".

Tone direction:
- Production copy should be calm, confident, and short.
- Avoid explaining the system architecture in UI text.
- Avoid overusing AI language where the user cares about outcome.
- Say what the user can do next.


COLOR REQUIRED CHANGES

Current issue:
- The UI uses many tinted panels, gradients, colored badges, and decorative accents.
- This makes the dashboard feel busy and reduces the meaning of status colors.

Required changes:
- Define a smaller status palette:
  - Draft: neutral
  - Review: blue
  - Scheduled: purple or indigo
  - Published: green
  - Error/blocked: red
  - Needs refresh: amber
- Reserve strong color for status and primary action.
- Make background panels mostly neutral.
- Reduce gradient usage in dashboard pages.
- Ensure contrast passes accessibility for small labels and muted text.
- Avoid large one-color page themes.
- Make disabled states visibly disabled but still readable.
- Use consistent border colors across all cards and panels.


SIZE, SPACING, AND CONTAINER REQUIRED CHANGES

Current issue:
- Boxes and cards are often too large for the amount of content inside them.
- Some pages use nested padding from both dashboard layout and AI Blogger layout.
- Many cards use large rounded corners that feel oversized for admin UI.

Required changes:
- Create one shared AI Blogger page shell for all dashboard routes.
- Standardize page max width and horizontal padding.
- Avoid double padding from nested layouts.
- Use 8px to 12px radius for normal cards and panels.
- Use 16px radius only for larger feature panels if necessary.
- Reduce vertical spacing between related controls.
- Increase spacing between unrelated sections.
- Use stable widths for buttons and icon controls.
- Ensure button text never clips.
- Ensure badges do not overflow their containers.
- Keep form fields aligned and grouped.
- Use tables/lists where many repeated items exist.
- Avoid nested card structures unless the inner card is a real repeated item.


TYPOGRAPHY REQUIRED CHANGES

Current issue:
- Many sections use small uppercase labels and dense microcopy.
- Editor content and dense cards are not always comfortable to read.
- Some headings feel too large for compact admin panels.

Required changes:
- Use clear hierarchy: page title, section title, field label, helper text.
- Keep helper text short.
- Avoid all-caps labels for important information.
- Increase readability for article/editor body content.
- Reduce heading size inside compact cards.
- Keep metadata text readable on mobile.
- Do not rely on tiny text for important warnings or setup instructions.


ACCESSIBILITY REQUIRED CHANGES

Required changes:
- Add aria-label to every icon-only link and button.
- Do not rely only on title attributes for tooltips.
- Ensure keyboard focus states are visible.
- Make mobile navigation understandable to screen readers.
- Connect form errors to fields with accessible descriptions.
- Provide error summaries for long forms.
- Ensure color is not the only status indicator.
- Check contrast for muted text, badges, and disabled controls.
- Ensure loading states announce meaningful progress where needed.
- Ensure modal, dropdown, and tab interactions are keyboard accessible.


EMPTY, LOADING, AND ERROR STATES

Required changes:
- Use route-specific skeletons instead of generic spinners.
- Every empty state must answer:
  - What happened?
  - Why is this empty?
  - What can I do next?
- Search Console disabled state should lead to setup.
- No posts state should lead to generate draft.
- No clusters state should lead to generate or assign clusters.
- No refresh candidates should explain the data requirement.
- Super-admin unavailable states should be polished and role-aware.
- Errors should be inline, recoverable, and written in user language.


INTERACTION REQUIRED CHANGES

Required changes:
- Add sticky primary actions on long forms and mobile pages.
- Add dirty-state handling for settings and editor forms.
- Add confirmations for destructive actions and publishing actions.
- Add clear success toasts after save, sync, generate, and publish.
- Keep hover states subtle and consistent.
- Avoid layout shifts when filters, badges, or loading text change.
- Use accordions/tabs for secondary information.
- Make bulk actions visible only when items are selected.


PERFORMANCE AND PERCEIVED SPEED

Current issue:
- The post detail route can feel slow or heavy because it loads many supporting data sources.
- Some client-loaded admin pages show a plain spinner before meaningful content.

Required changes:
- Add progressive skeletons.
- Defer heavy SEO, SERP, and performance panels until the editor shell is visible.
- Cache or prefetch supporting data where appropriate.
- Keep first meaningful paint focused on the page title, status, and primary content.
- Avoid blocking the whole page for secondary audit data.
- Profile post detail loading.

FINAL PRODUCTION STANDARD
Before calling AI Blogger production-ready, each page should pass this checklist:
- No internal roadmap or implementation copy is visible.
- Every empty state has a clear next action.
- Mobile navigation is labeled and accessible.
- Cards, buttons, badges, and form fields do not clip or overflow.
- Repeated content is shown in compact lists or tables.
- Colors communicate status instead of decoration.
- Containers use consistent radius, padding, and max widths.
- The primary action on every page is obvious.
- Loading states show the page structure, not just a spinner.
- The product feels like a reliable content operations tool, not a demo.
