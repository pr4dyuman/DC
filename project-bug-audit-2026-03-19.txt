Project Bug Audit
Date: 2026-03-19
Repository: agency-os

Scope
- Repository-wide static scan plus manual review of high-risk flows (auth, uploads, tasks, assets, finance, chat, and Singularity).
- Verification run:
  - npm run build -> passed
  - .\node_modules\.bin\eslint.cmd . -> passed
  - .\node_modules\.bin\tsc.cmd --noEmit -> passed
- This file lists confirmed bugs found in the current code. It is not a guarantee that no additional bugs exist.

1. High - Large direct uploads bypass trial, storage, and file-content protections
Files:
- components/projects/AddAssetModal.tsx:173-185
- app/api/upload-blob/route.js:30-59
- app/api/upload-dc/route.js:147-229
Problem:
- Files larger than 4 MB are routed from the browser directly to /api/upload-blob.
- The direct-upload handler only checks auth, extension, and allowed content types before minting the token.
- The normal /api/upload-dc path also enforces trial-expiration checks, agency storage limits, image magic-byte validation, image dimension checks, and usage.storage accounting.
Impact:
- Large uploads can bypass plan enforcement and storage-accounting logic.
- Security checks differ by file size, which makes the safer path optional.
Suggested fix:
- Reuse the same validation and accounting logic in /api/upload-blob before token generation and after upload completion, or route all uploads through a shared validator.

2. Medium - Wrong MIME type blocks valid .xlsx direct uploads
Files:
- app/api/upload-blob/route.js:43-49
- app/api/upload-dc/route.js:31
Problem:
- The direct-upload handler allows application/vnd.openxmlformats-officedocument.spreadsheetml.document instead of the correct spreadsheetml.sheet MIME.
Impact:
- Legitimate .xlsx uploads can fail on the direct-upload path even though small-file uploads accept them.
Suggested fix:
- Replace the direct-upload MIME entry with application/vnd.openxmlformats-officedocument.spreadsheetml.sheet.

3. High - Missing user-permission records grant broad task powers
Files:
- lib/actions/task-queries.ts:13-19
- lib/actions/task-queries.ts:46-50
- lib/types-business.ts:244-249
Problem:
- Server-side fallback permissions default to canManageTasks=true, canMarkDone=true, deleteAccess="any".
- The shared default in lib/types-business.ts says deleteAccess="own", so even the defaults are inconsistent.
- Any user without an explicit permissions entry inherits broad task-management rights.
Impact:
- A missing settings record becomes a privilege grant instead of a safe deny/default-minimum behavior.
Suggested fix:
- Change the server fallback to least-privilege defaults and keep the shared/server defaults identical.

4. High - Task mutation actions do not verify project/task access
Files:
- lib/actions.ts:767-818
- lib/actions/task-mutations.ts:229-279
- lib/actions/task-mutations.ts:281-346
- lib/actions/task-collaboration.ts:18-36
- lib/actions/task-collaboration.ts:57-82
Problem:
- deleteTask, updateTaskStatus, updateTask, and addComment require auth and permission flags, but they do not verify that the caller can access the task's project.
- The mutation implementations only scope by taskId + agencyId and permission flags.
Impact:
- A user who knows a task ID can mutate or comment on tasks outside their intended project scope within the same agency.
- This is especially risky because task IDs are already exposed in normal project pages.
Suggested fix:
- Resolve the task first, then enforce canCurrentUserAccessProject(task.projectId, agency.id) or an equivalent task-level authorization check in every mutation path.

5. High - Project task UI exposes edit/create/status controls to users who inherit permissive defaults
Files:
- app/dashboard/projects/[slug]/page.tsx:27-34
- components/projects/ProjectView.tsx:75-78
- components/projects/KanbanBoard.tsx:130-178
- components/projects/ViewTaskModal.tsx:92-95
- components/projects/EditTaskModal.tsx:93-97
Problem:
- The project page always loads getUserPermissions(currentUser.id) and passes those permissions into task UI.
- The UI enables create, edit, delete, and status-change affordances whenever canManageTasks/canMarkDone/deleteAccess allow it.
- Because the server fallback is permissive, users with no explicit permissions can receive management UI.
Impact:
- Clients or employees can be shown task-management controls they should not have, which lines up with the server-side authorization gap above.
Suggested fix:
- Tighten the default permission model and make the UI fail closed instead of using permissive nullish fallbacks.

6. High - Employee asset uploads are not checked against project access
Files:
- lib/actions.ts:1111-1122
- lib/actions/assets.ts:22-59
Problem:
- getProjectAssets checks canCurrentUserAccessProject before returning assets.
- addProjectAsset only requires role admin/manager/employee and then writes the asset without checking whether the caller can access asset.projectId.
Impact:
- An employee can tamper the client request and upload assets into another project within the same agency.
Suggested fix:
- Add canCurrentUserAccessProject(asset.projectId, agency.id) to addProjectAsset before calling addProjectAssetImpl.

7. Medium - Asset/activity audit trail records uploads as "Admin" regardless of uploader
Files:
- components/projects/AddAssetModal.tsx:247-255
- components/projects/AddAssetModal.tsx:294-302
- lib/actions/assets.ts:48-52
Problem:
- Both file mode and link mode submit uploadedBy: "Admin".
- The server persists that string and uses it in activity logging.
Impact:
- Activity logs and asset metadata misidentify the real uploader.
Suggested fix:
- Populate uploadedBy on the server from the authenticated caller instead of trusting the client payload.

8. Medium - Binary files are read as text before upload
Files:
- components/projects/AddAssetModal.tsx:121-133
Problem:
- Any selected file whose inferred type stays "file" or "code" is read with FileReader.readAsText().
- That includes binary formats like PDF, DOCX, XLSX, etc.
Impact:
- Large binary files can create garbage content strings, increase browser memory usage, and later bloat stored asset documents.
Suggested fix:
- Only read known text/code formats as text. Skip content extraction for binary files or move extraction to a safe server-side parser.

9. Medium - Asset content is effectively unbounded
Files:
- components/projects/AddAssetModal.tsx:255
- components/projects/AddAssetModal.tsx:302
- lib/actions/assets.ts:27-31
- lib/mongodb.ts:277-285
Problem:
- The client submits formData.content for assets.
- The server sanitizes name/description/url but does not limit or sanitize content.
- The schema stores content as a plain String with no size guard.
Impact:
- Large text payloads can bloat MongoDB documents and may eventually hit document-size limits.
Suggested fix:
- Add server-side length limits and content-type-aware processing before persisting content.

10. Medium - Forgot-password flow burns a valid OTP before validating the new password
Files:
- app/api/forgot-password/reset/route.ts:45-67
- app/api/signup/send-otp/route.ts:153-180
Problem:
- The reset route verifies the OTP first.
- verifyOtp deletes the OTP record on success.
- Only after that does the route validate password strength/policy.
Impact:
- A user who enters a weak or invalid new password loses a valid OTP and must request another code.
Suggested fix:
- Validate the new password before consuming the OTP, or make OTP consumption atomic with the password update.

11. Medium - Contact submissions can pass the required-field check with an invalid phone and still be saved
Files:
- app/api/contact-dc/route.js:41-47
- app/api/contact-dc/route.js:59-82
Problem:
- phone is checked for truthiness before sanitization.
- sanitizePhone(phone) can turn bad input into an empty string, but there is no post-sanitization phone validation.
Impact:
- Invalid contact requests can be stored with blank/invalid phone numbers.
Suggested fix:
- Re-validate phone after sanitization and reject empty/invalid results.

12. Medium - Failed chat sends leave a phantom optimistic message in the UI
Files:
- components/chat/useChatPanelState.ts:113-142
Problem:
- handleSend appends an optimistic temp message before awaiting sendMessage().
- On failure it only shows a toast and never removes or rolls back the optimistic entry.
Impact:
- Users can see messages that were never actually stored or delivered.
Suggested fix:
- Remove the temp message on failure or reconcile optimistic entries with server-confirmed messages.

13. Medium - Project status auto-completes but never reopens when work returns
Files:
- lib/actions/task-effects.ts:143-151
Problem:
- When the last open task becomes Done, the project is set to Completed.
- There is no corresponding reverse transition when a task later moves out of Done.
Impact:
- Projects can remain marked Completed while open tasks still exist.
Suggested fix:
- Add reverse status logic when a task leaves Done or when any open task is reintroduced.

14. Medium - Task forms allow "Unassigned" even though the schema requires assigneeId
Files:
- components/projects/CreateTaskModal.tsx:93-103
- components/projects/CreateTaskModal.tsx:153-157
- components/projects/EditTaskModal.tsx:132-135
- lib/mongodb.ts:169-175
- lib/actions/task-mutations.ts:118-121
- lib/actions/task-mutations.ts:307-340
Problem:
- The UI allows an empty assigneeId via an "Unassigned" option.
- The Task schema marks assigneeId as required.
- The mutation code only validates assignee existence when assigneeId is truthy, so empty strings slip past those checks.
- updateTaskImpl writes with updateOne and no validator enforcement.
Impact:
- Task creation can fail unexpectedly, and edits can persist invalid/unexpected assignee state.
Suggested fix:
- Either make assigneeId optional everywhere or remove the Unassigned option and enforce a required assignee consistently.

15. Medium - Client presence stays stale when reading messages
Files:
- components/chat/useChatPanelState.ts:62-66
- lib/chat.ts:85-106
- lib/chat.ts:256-257
- lib/chat.ts:264-280
Problem:
- markAsRead is called when a conversation is opened, but markAsRead only updates UserModel.lastActiveAt.
- heartbeat can update ClientModel, but it is only called from sendMessage.
Impact:
- Clients who open/read conversations without sending messages may never get a fresh lastActiveAt value.
Suggested fix:
- Update both UserModel and ClientModel in markAsRead, or call a shared presence updater on message reads and polling activity.

16. High - Singularity read tools expose agency-wide finance data to employees
Files:
- app/dashboard/singularity/page.tsx:17-22
- lib/singularity-route-helpers.ts:6-19
- lib/singularity-route-helpers.ts:108-113
- lib/singularity-tool-def-read.ts:51-66
- lib/singularity-tool-def-read.ts:98-106
- lib/singularity-tool-read.ts:131-140
- lib/singularity-tool-read.ts:233-239
- lib/actions.ts:903-907
- lib/actions.ts:960-964
- lib/actions/finance-queries.ts:39-47
- lib/actions/finance-queries.ts:121-139
Problem:
- Employees can reach the Singularity page as long as they have AI access.
- Read-only tools like get_transactions and get_invoices are not mapped in TOOL_PERMISSION_FLAGS, so filterToolsByPermissions lets them through.
- Those tool handlers call getTransactions/getInvoices.
- The finance query layer restricts clients, but not employees, so an employee can retrieve agency-wide finance/invoice data.
Impact:
- AI-enabled employees can query sensitive financial data they may not be authorized to view in the normal UI.
Suggested fix:
- Add finance read tools to the AI permission map and/or enforce role checks inside getTransactions/getInvoices for employee callers.

Summary
- Static checks are clean, but there are still confirmed runtime/security/data-integrity bugs in the current code.
- The highest-risk areas are uploads, task authorization, asset writes, and Singularity finance access.
